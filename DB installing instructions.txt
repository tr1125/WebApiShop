הוראות התקנת הDB:

הנה המדריך המפורט לשחזור מסד הנתונים (Restore) במחשב היעד, שלב אחר שלב, כדי לוודא שכל התוכן עובר בצורה מושלמת:

שלב 1: הכנת הקובץ העתק את קובץ ה-.bak שיצרת במחשב המקור אל מחשב היעד.

טיפ חשוב: מומלץ להניח את הקובץ בתיקיית הגיבויים כברירת מחדל של SQL Server (בדרך כלל בנתיב C:\Program Files\Microsoft SQL Server\MSSQL... \Backup), זה מונע בעיות של הרשאות גישה לקובץ.

שלב 2: פתיחת אשף השחזור ב-SSMS פתח את SQL Server Management Studio במחשב החדש והתחבר לשרת.

לחץ קליק ימני על תיקיית Databases בסייר האובייקטים (Object Explorer).

בחר באפשרות Restore Database....

שלב 3: בחירת המקור (Source) בחלון שנפתח, תחת הסעיף Source, סמן את האפשרות Device.

לחץ על כפתור שלוש הנקודות (...) שנמצא בצד ימין.

בחלונית הקטנה שנפתחה, ודא שסוג הקובץ הוא "File" ולחץ על Add.

נוט בתיקיות עד שתמצא את קובץ ה-.bak שהעברת, בחר אותו ולחץ על OK.

לחץ שוב על OK בחלונית בחירת המכשיר. כעת המערכת תסרוק את הגיבוי ותציג אותו ברשימה למטה.

שלב 4: הגדרות חשובות (Files & Options) לפני שאתה לוחץ על אישור סופי, כדאי לעבור על שתי לשוניות בצד שמאל:

לשונית Files: ודא שסומן ה-V בתיבת Relocate all files to folder. זה חשוב מאוד אם נתיבי התיקיות במחשב החדש שונים מהמחשב הישן (למשל שם משתמש שונה ב-Windows).

לשונית Options:

אם כבר קיים DB בשם הזה ואתה רוצה לדרוס אותו, סמן Overwrite the existing database (WITH REPLACE).

וודא שסומן Close existing connections to destination database (כדי שהשחזור לא ייכשל בגלל שמישהו "תופס" את מסד הנתונים).

שלב 5: סיום לחץ על OK בתחתית החלון.

המתן לסיום התהליך. בסיום תופיע הודעה: Database 'YourDBName' restored successfully.

מה השלב הבא? לאחר השחזור, כדאי לרענן (Refresh) את תיקיית ה-Databases ב-SSMS. כעת תוכל לפתוח את הטבלאות ולראות שכל המידע שם.

היי! אני מבין לגמרי את התסכול. לעבוד על פרויקט משני מוקדים כשבאחד מהם הגישה מוגבלת זה אתגר לא פשוט, במיוחד כשמערכת ה-SSMS (SQL Server Management Studio) מתנהגת כאילו היא חיה בעולם מקביל ולא מזהה את התיקיות שאתה רואה בסייר הקבצים.

הסיבה שאתה רואה "כונן D" שונה בתוך ה-SSMS היא לרוב עניין של הרשאות שירות (Service Permissions). מנוע ה-SQL Server רץ תחת משתמש מערכת ספציפי, והוא לא תמיד רואה את מה שהמשתמש האישי שלך רואה.

הנה כמה פתרונות שיכולים לעזור לך לעקוף את המגבלה הזו בלי להזדקק להרשאות מנהל מורכבות:

1. שימוש בתיקיית ה-Public (הנתיב הכי פחות חסום)
במערכות עם הרשאות מוגבלות, תיקיית ה-Public בדרך כלל פתוחה לקריאה ולכתיבה עבור כל שירותי המערכת.

נסה להעתיק את קובץ ה-.bak לנתיב: C:\Users\Public\Documents.

בתוך ה-SSMS, כשאתה מחפש את הקובץ ל-Restore, נווט לנתיב הזה באופן ידני. רוב הסיכויים שה-SQL Server יצליח "לראות" את הקובץ שם.

2. שימוש בקיצור דרך ישיר (Manual Path)
לפעמים ה-UI של SSMS פשוט לא מצליח לרענן את רשימת התיקיות. במקום לדפדף:

בחלון בחירת הקובץ (Locate Database File), נסה להדביק או להקליד את הנתיב המלא של הקובץ ישירות בשורת הטקסט למטה.

ודא שנתת לקובץ הרשאות "Everyone" (קליק ימני על הקובץ -> Properties -> Security -> Edit -> Add -> כתיבת "Everyone" ואישור).

3. פתרון ה-T-SQL (עוקף ממשק ויזואלי)
אם הממשק הגרפי עושה בעיות, פקודת SQL ישירה לפעמים מצליחה לדלג על מגבלות התצוגה של הסייר. פתח "New Query" והרדץ את הקוד הבא (כמובן, שנה את השמות והנתיבים בהתאם):

SQL
USE [master]
RESTORE DATABASE [YourDatabaseName] 
FROM DISK = N'C:\Path\To\Your\File.bak' 
WITH MOVE 'Logical_Data_Name' TO 'C:\Path\To\New\DataFile.mdf',
MOVE 'Logical_Log_Name' TO 'C:\Path\To\New\LogFile.ldf',
REPLACE, STATS = 5
טיפ: כדי לדעת מהם ה-Logical Names, תוכל להריץ קודם:
RESTORE FILELISTONLY FROM DISK = N'C:\Path\To\Your\File.bak'

4. האופציה העננית (אם יש אינטרנט)
אם כלו כל הקיצין והמחשב חסום הרמטית, אולי כדאי לשקול להשתמש בדאטה-בייס מרוחק (כמו Azure SQL או פתרון חינמי אחר ברשת). כך לא תצטרך לסנכרן קבצים ידנית – שני המחשבים פשוט יתחברו לאותה כתובת IP ונתונים יישארו זהים בזמן אמת.
